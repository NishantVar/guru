# Lesson Writing Principles

Use this as the default instruction set when an AI writes lessons for The New Order.

This document is intentionally short. It does not try to preserve all background research. It keeps only the principles that should actually change how a lesson gets written.

## Primary Goal

The goal is to develop self-sufficient AI practitioners who can use AI to improve real work, automate workflows, build useful systems, and reason from first principles.

A lesson succeeds when the learner leaves more capable, not just more informed or entertained.

## Core Principles

### 1. Lead with the capability

Open by saying what the lesson teaches, why it matters in real work, and what the learner will be able to do after it.

Do not open with history, definitions, or architecture.

### 2. Teach concrete -> mechanism -> judgment

Start with what the thing does in practice.

Then explain how it works at the simplest useful level.

Then explain when to use it, where it breaks, and what good judgment looks like.

### 3. Keep the scope tight

Each lesson should have one core idea and no more than 3-4 supporting ideas.

If the lesson cannot be summarized in about 20 plain words, it is probably trying to do too much.

Cut adjacent interesting material rather than cramming it in.

### 4. Give the learner an early win

Within the first third, the learner should understand, do, or notice something concrete.

They should quickly get an "I can follow this" moment.

Do not spend half the lesson on theory before any payoff appears.

### 5. Make the learner do something

Include at least one active element.

- Try a prompt
- Inspect an output
- Answer a question
- Predict what will happen
- Fix a broken example
- Apply the idea to their own workflow

Passive reading alone is not enough.

### 6. Prefer durable understanding over tool trivia

Teach principles that survive model, product, and interface changes.

Use current tools as examples, not as the curriculum itself.

Whenever possible, show how the idea transfers across tasks and tools.

### 7. Use simple language without becoming misleading

Prefer observable examples, plain English, and everyday comparisons.

Define jargon on first use.

Use analogies carefully. They should clarify the concept, not replace it.

If an analogy breaks down in an important way, say so.

### 8. Build confidence before nuance

Give the learner a stable first mental model before adding caveats.

Do not front-load edge cases, exceptions, or expert debates.

But do be honest when something is genuinely uncertain or contested.

### 9. Write for Discord

Keep sections short and easy to scan.

Frontload the first sentence of each section.

Use headings, bullets, examples, and code blocks to reduce friction.

Prefer multiple small blocks over one wall of text.

### 10. End with momentum

Close with a short recap, one concrete next step, and the next lesson, exercise, or question.

A learner should never finish a lesson and wonder what to do next.

## Anti-Patterns

- Opening with background instead of learner payoff
- Explaining several concepts when one lesson should focus on one
- Using jargon as a shortcut instead of teaching
- Treating analogies as literal truth
- Writing to sound smart instead of to make the learner capable
- Using hype, mysticism, or "AI magic" framing
- Ending without practice or a clear next step

## Pre-Publish Check

Before finalizing a lesson, confirm:

1. Can the lesson's value be stated in one sentence?
2. Does the first paragraph say what the learner will get and why it matters?
3. Is there a quick win early?
4. Is there at least one active step?
5. Can a non-technical learner follow without unexplained jargon?
6. Does the lesson move from concrete example to mechanism to judgment?
7. Will the lesson still make sense if today's tool changes next month?
8. Is the formatting easy to scan in Discord?
9. Does the lesson end with recap plus next step?

If two or more answers are "no," rewrite before publishing.
